# /plan Skill

当用户提及 "Plan X"、"继续 Plan X"、"开始 Plan X" 或类似计划式开发指令时：

## 执行步骤

1. **检查现有计划**
   - 搜索项目中的计划文件：`docs/plans/`、`plans/`、根目录 markdown、或任何包含 "Plan" 的文档
   - 读取 `PLAN_STATUS.md`（如果存在）了解当前计划全景

2. **若计划已存在**
   - 总结该计划的当前状态（已完成、进行中、阻塞）
   - 列出该计划涉及的关键文件和最近的修改
   - 询问用户："Plan X 当前状态如上，确认继续执行还是调整范围？"

3. **若计划不存在（关键）**
   - **禁止仅报告"Plan X 不存在"然后停止**
   - 基于已完成计划的演进规律、git 历史、项目结构，推断 Plan X 最可能的目标
   - 主动询问："Plan X 尚未创建。基于当前项目上下文，建议包含以下内容：
     - 目标：[推断目标]
     - 涉及文件：[推断文件]
     - 验收标准：[推断标准]
     您希望如何定义？或者我现在就创建草案？"

4. **创建计划草案（如用户同意）**
   - 使用与用户已有计划相同的格式
   - 必须包含：目标、涉及文件/模块、实施步骤（带勾选框）、验收标准、验证步骤
   - 写入 `PLAN_STATUS.md` 或 `docs/plans/plan-x.md`
   - 确认后再开始实施

## 禁止行为
- 仅说 "Plan X doesn't exist" 然后等待用户下一步指令
- 在没有确认的情况下直接修改代码声称是 Plan X 的一部分
- 忽略已有计划的历史上下文，孤立地看待新计划
