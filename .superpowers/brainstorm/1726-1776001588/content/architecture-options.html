三种架构方案对比

核心约束：10分钟长任务 + 10–50 并发用户 + 公网 SaaS + Supabase 生态

A

Next.js + Supabase 全栈（推荐）

前端 Vercel · 后台 Worker 独立部署 · pg-boss 任务队列 · Supabase Realtime 进度推送

浏览器 → Next.js (Vercel) → Supabase DB + pg-boss 任务队列 → Worker 服务 (Railway/Fly.io) → 外部 AI API × N
实时进度： Supabase Realtime 订阅任务状态表 → 动态工作流 UI 更新

优势

  • Supabase 生态高度整合（Auth/DB/Storage/Realtime 一体）
  • pg-boss 直接跑在 PostgreSQL 内，无需额外 Redis
  • Worker 无执行时长限制，天然支持10分钟任务
  • Next.js + Vercel 部署简单，前端开发体验好
  • Supabase Realtime 天然支持动态工作流进度推送

权衡

  • Worker 需单独部署一个服务（Railway 约 $5/月）
  • 首次搭建比纯 Edge Functions 方案稍复杂
B

Next.js + Supabase Edge Functions（纯 Serverless）

前端 Vercel · Edge Functions 编排 · 无独立 Worker

浏览器 → Next.js → Supabase Edge Functions（编排） → 外部 AI API × N（并发）
⚠️ Edge Functions 最长执行时间： 150 秒（2.5 分钟）

优势

  • 完全 Serverless，无需维护 Worker 服务
  • 部署极简，成本最低

硬限制

  • ❌ Edge Functions 最长 150s，无法支持 10 分钟任务
  • 需要切分任务+轮询，架构变复杂且不可靠
  • 不推荐用于本系统
C

Next.js + 独立 Express 后端 + BullMQ（Redis）

前端 Vercel · 独立 API 服务 · Redis 任务队列 · WebSocket 实时进度

浏览器 → Next.js → Express API + BullMQ Worker → Redis 队列 → 外部 AI API × N
Supabase 仅用于：Auth + 数据存储

优势

  • BullMQ + Redis 是业界成熟的任务队列方案
  • WebSocket 支持最灵活的实时通信
  • 技术栈可控性最强

权衡

  • 需额外维护 Redis 服务（增加运营成本和复杂度）
  • Supabase 的 Realtime 优势无法充分利用
  • 部署和维护工作量最大
推荐：方案 A — 在 Supabase 生态内实现完整功能，pg-boss 解决长任务问题，Realtime 驱动动态工作流看板，Worker 单独部署解决10分钟超时限制，整体复杂度适中。
